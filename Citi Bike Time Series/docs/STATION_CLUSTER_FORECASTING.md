# Station-Cluster Forecasting

This note explains the operational forecasting layer generated by:

```bash
python src/citibike_station_cluster_forecast.py --start-month 2024-01 --end-month 2024-12 --skip-download
```

The purpose is to move the project from an aggregate demand proof to a
station-aware planning case study. The decision target is:

```text
rebalancing and station-capacity planning
```

The model still forecasts trip starts, not bike inventory. That distinction is
important: trip starts identify demand pressure, but a production rebalancing
system would also need dock capacity, live availability, truck moves, and
service-level targets.

## What This Adds

| Area | Full-Year Proof | Station-Cluster Layer |
| --- | --- | --- |
| Target | Systemwide hourly trip starts | Systemwide plus station-cluster hourly trip starts |
| Station key | Station ID/name ranking | Stable station IDs with observed names and coordinates |
| Groups | None | 6 deterministic geographic clusters from top 120 station IDs |
| Exogenous features | Federal holidays only | Weather, holidays, and major public event windows |
| Models | Previous-week and calendar-lag ridge among others | Previous-week, calendar-lag ridge, and weather/event ridge |
| Decision | General forecast proof | Rebalancing and capacity-planning watchlist |

## Source Design

The script streams the same public Citi Bike archives used by the full-year
proof:

```text
https://s3.amazonaws.com/tripdata/YYYYMM-citibike-tripdata.zip
```

It also uses hourly New York City weather from the Open-Meteo historical
archive API:

```text
https://archive-api.open-meteo.com/v1/archive
```

Raw Citi Bike ZIP files are cached under `work/data/multi_month/`. Weather JSON
is cached under `work/data/weather/`. Both paths are ignored by Git.

## Station Metadata And Clustering

Station metadata is derived from observed trip-history fields:

- `start_station_id`
- `start_station_name`
- `start_lat`
- `start_lng`

The script keeps station IDs as the stable key, chooses the most common observed
station name for readability, and averages observed coordinates after removing
obvious out-of-market coordinates.

## Weather And Event Features

The weather/event ridge model uses the same leakage-aware calendar and lag
features as the full-year ridge baseline, then adds:

- temperature
- precipitation
- snowfall
- wind speed
- weather code
- precipitation and snow flags
- temperature-below-comfort and temperature-above-comfort transforms
- federal holiday windows
- holiday eve and holiday window flags
- selected major public event windows
- event peak-hour flags

The event calendar is compact and transparent. It includes U.S. federal
holidays plus a small 2024 public-event feature set:

- NYC Pride March
- TCS New York City Marathon
- Macy's Thanksgiving Day Parade
- New Year's Eve

This is not a complete NYC event database. It is a portfolio feature layer that
shows how external context can enter rolling validation.

## Validation Design

The script keeps the same rolling validation discipline as the full-year proof:

```text
train = rows where hour < origin
test = rows where origin <= hour < origin + 24 hours
```

Current settings:

| Setting | Value |
| --- | ---: |
| Training warmup | 60 days |
| Origin cadence | weekly |
| First origin | 2024-03-01 |
| Last origin | 2024-12-27 |
| Origins per segment | 44 |
| Forecast horizon | 24 hours |

Models compared:

| Model | Definition |
| --- | --- |
| Previous week | Forecast equals the value from 168 hours earlier |
| Calendar + lag ridge | Regularized linear model with calendar cycles and prior-demand features |
| Weather/event ridge | Calendar + lag ridge features plus weather, holiday, and event features |

Observed historical weather is used as a stand-in for weather forecast inputs.
In production, these fields should come from forecasts available before each
origin.

## Decision Implication

The right portfolio takeaway is not just "the model improved." It is:

> Use station-cluster forecasts as a rebalancing and capacity-planning
> watchlist, while retaining calendar-lag ridge as the control benchmark.

The top planning priority in the current run is:

| Priority Cluster | Annual Starts | Weather/Event MAE | Weather/Event WAPE | Lift vs Previous Week |
| --- | ---: | ---: | ---: | ---: |
| Cluster 01 - W 21 St & 6 Ave area | 3,971,209 | 71.6 | 14.3% | +21.1% |

This cluster combines the highest annual demand, meaningful forecast error, and
clear improvement over the previous-week operating baseline.

## Interview Defense

The strongest way to explain this layer:

> I moved the project from aggregate demand to an operational grain. I built
> stable station groups from station IDs and observed metadata, then evaluated
> aggregate and station-cluster forecasts in the same rolling-origin framework.
> Weather and event features were only treated as useful because they beat both
> previous-week and calendar-lag ridge baselines out of sample.

## What It Still Does Not Prove

- It does not forecast live bike or dock availability.
- It does not know station capacity.
- It does not use rebalancing truck logs or service-level outcomes.
- It uses observed historical weather as a stand-in for weather forecasts.
- The event calendar is intentionally compact, not exhaustive.
- It does not produce calibrated prediction intervals.

## Strong Next Step

The next best portfolio upgrade is not "add more features." It is to add
operations context:

1. Join station capacity and live availability/status snapshots.
2. Score whether cluster demand forecasts predict stockout or full-dock risk.
3. Add a calibrated alert threshold for the rebalancing watchlist.
4. Evaluate errors by commute peak, weather pressure, and event windows.
5. Convert the report into a lightweight dashboard or notebook walkthrough.

# Multi-Month Forecast Proof

This note explains the full-2024 proof layer generated by:

```bash
python src/citibike_multi_month_proof.py --start-month 2024-01 --end-month 2024-12
```

The purpose is to move the project beyond a one-month winter showcase. The
January report still proves the methods; this layer proves the project can
handle a year of public data and a repeated validation design.

## What This Adds

| Area | January Showcase | Full-2024 Proof |
| --- | --- | --- |
| Coverage | Jan. 1-31, 2024 | Jan. 1-Dec. 31, 2024 |
| Raw rows read | 1.89M | 44.30M |
| Valid rows | 1.89M | 44.25M |
| Hourly observations | 744 | 8,784 |
| Missing hours | 0 | 0 |

## Source Design

The script streams public Citi Bike monthly archives from:

```text
https://s3.amazonaws.com/tripdata/YYYYMM-citibike-tripdata.zip
```

Raw archives are cached under:

```text
work/data/multi_month/
```

That directory is intentionally ignored by Git. The committed artifacts are the
aggregated outputs and charts under `outputs/`.

## Quality Filter

A row is counted as valid only when:

1. `started_at` parses.
2. `ended_at` parses.
3. Trip duration is between 1 and 240 minutes.

The full-2024 run read 44,303,209 raw rows and retained 44,247,816 valid rows,
for a 99.87% valid rate.

## Forecast Target

The target is:

```text
valid Citi Bike trip starts per hour
```

The generated full-year hourly panel contains 8,784 rows, which matches every
hour in leap year 2024. The missing-hour count is 0.

## Validation Design

The full-year proof uses rolling 24-hour forecast origins.

Current configuration:

| Setting | Value |
| --- | ---: |
| Training warmup | 60 days |
| Origin cadence | weekly |
| First origin | 2024-03-01 |
| Last origin | 2024-12-27 |
| Origins scored | 44 |
| Horizon per origin | 24 hours |
| Holdout hours per model | 1,056 |

The leakage boundary is simple:

```text
train = rows where hour < origin
test = rows where origin <= hour < origin + 24 hours
```

Every model is fit or profiled using only rows before the origin.

## Models Compared

| Model | Definition |
| --- | --- |
| Previous day | Forecast equals the value from 24 hours earlier |
| Previous week | Forecast equals the value from 168 hours earlier |
| Hour-of-day profile | Expanding training average for the same hour of day |
| Weekday/hour profile | Expanding training average for the same weekday and hour |
| Seasonal lag blend | 65% weekday/hour profile plus 35% previous-week forecast |
| Calendar + lag ridge | Regularized linear model using calendar features and prior-demand features |

The ridge model uses only features available before the 24-hour forecast
horizon:

- hour sine/cosine
- day-of-week sine/cosine
- month sine/cosine
- weekend flag
- federal holiday flag
- days since panel start
- lag 24 hours
- lag 168 hours
- 24-hour rolling mean anchored one day back
- 168-hour rolling mean anchored one day back

It does not use weather or event data in this version.

## Current Results

| Model | MAE | RMSE | MAPE | Origin Win Rate |
| --- | ---: | ---: | ---: | ---: |
| Calendar + lag ridge | 750 | 1,163 | 28.1% | 36.4% |
| Previous week | 918 | 1,538 | 22.9% | 36.4% |
| Previous day | 1,034 | 1,547 | 23.4% | 11.4% |
| Seasonal lag blend | 1,350 | 1,831 | 29.0% | 9.1% |
| Weekday/hour profile | 1,792 | 2,369 | 36.7% | 6.8% |
| Hour-of-day profile | 1,970 | 2,550 | 38.5% | 0.0% |

MAE is the primary comparison because it is easiest to explain as rides per
hour. MAPE is included, but it can favor models differently when low-volume
overnight hours create large percentage swings.

## Interview Defense

The strongest way to explain this layer:

> I first built a clean one-month time-series profile so the methods were easy
> to review. Then I added a full-year proof layer that streams 44M public trip
> records, creates a complete hourly panel, and validates forecasts across 44
> rolling origins. I kept transparent baselines in the comparison, then added a
> simple regularized calendar-lag model to show lift without hiding behind a
> black box.

## What It Still Does Not Prove

- It does not forecast station inventory or bike availability.
- This aggregate full-year script does not include weather or event features;
  those are added in the station-cluster forecasting layer.
- It does not solve inventory-aware station forecasting; the station-cluster
  layer forecasts trip-start demand, not bike or dock availability.
- It does not produce calibrated prediction intervals.
- It does not select an operating decision by itself; the station-cluster layer
  ties the next model to rebalancing and station-capacity planning.

## Completed Next Step

The next portfolio upgrade is now implemented in:

```text
python src/citibike_station_cluster_forecast.py --start-month 2024-01 --end-month 2024-12 --skip-download
```

That layer builds stable station groups from station IDs and observed metadata,
forecasts the system total plus 6 station clusters, adds weather and
holiday/event features, evaluates the richer model against previous-week and
calendar-lag ridge baselines, and ties the result to rebalancing and
capacity-planning review.
